# 壁纸系统需求规格

## 一、概述

壁纸系统负责新标签页的壁纸展示。用户打开新标签页时，看到的第一件事就是壁纸——不能出现白屏、灰屏、闪烁。壁纸有两个来源：Bing 每日壁纸（默认）和用户自己上传的本地图片。

核心目标就三个：**打开不能闪白**、**切换要平滑**、**离线也能看**。

---

## 二、两种壁纸模式

| 模式 | 说明 | 谁来管 |
|------|------|--------|
| Bing 每日 | 每天从 Bing 获取一张新壁纸，自动缓存到本地，断网也能看昨天的 | 壁纸系统 |
| 本地轮换 | 用户上传的图片（最多 12 张），每次打开新标签页换一张 | 壁纸系统 + 本地图库 |

两种模式互斥，同一时刻只能是其中一种。默认是 Bing 模式。

**模式切换：**
- 用户第一次上传本地图片 → 自动切成「本地轮换」模式
- 用户删光所有本地图片 → 自动切回「Bing 每日」模式
- 用户在设置面板里点「重置」→ 立即切回 Bing 模式

---

## 三、零闪白显示 —— 双层架构

这是壁纸系统最核心的需求。原理很简单：**任何时候至少有一层显示着图片**。

页面上有两个叠在一起的图层：

| 图层 | 位置 | 职责 |
|------|------|------|
| 底层（back） | z-index 0 | 始终显示一张壁纸，常驻可见 |
| 前层（front） | z-index 1 | 做淡入过渡用的，平时完全透明（opacity: 0） |

**正常切换流程：**
1. 先把新图片加载到内存里，解码完成
2. 把新图片设到前层（此时前层还是透明的，用户看不到）
3. 让前层慢慢变不透明（0.5 秒淡入动画）
4. 动画结束后，把新图片同步到底层，前层清空恢复透明
5. 这样底层始终有图，用户只看到两张图平滑交替

**为什么底层不用动画：** 底层只是被赋值替换，不需要过渡效果。动画只发生在前层。

**关键约束：** CSS 动画时长必须是 500ms，代码里等 550ms（多 50ms 安全余量）再把图片同步到底层。

---

## 四、预加载 —— 第一帧就有图

这是整个页面最优先做的事情。在浏览器还没开始画第一帧之前，就要把壁纸设到页面上。

**加载顺序（严格）：**
1. HTML 中 `#wallpaperBack` 这个 div 必须出现在 `<script src="preload.js">` 之前
2. `preload.js` 同步执行——读 localStorage 里存的缩略图，直接设到底层背景
3. 浏览器画第一帧 → 壁纸已经在上面了

**预加载读什么（按优先级）：**

如果当前是「本地轮换」模式：
- 读 `ptab_local_index`（当前轮到第几张）、`ptab_img_order`（图片顺序）、`ptab_img_thumbs`（缩略图库）
- 算出当前该显示哪张，从缩略图库里取出对应的缩略图，赋给底层

如果当前是「Bing 每日」模式：
- 先读 `ptab_preview_thumb`（下一张壁纸的预计算缩略图——这是本地模式用的，但本地模式轮换时预先写好了）
- 如果没有，回退读 `ptab_bing_thumb`（Bing 壁纸的缩略图）

**全都没读到怎么办：** 极少见的情况——首次安装、没有任何缓存。底层为空，显示备用的深蓝渐变背景。不过用户几乎看不到，因为 `newtab.js` 加载后会立刻去获取壁纸。

## 五、Bing 每日壁纸

### 5.1 获取策略

同时请求两个服务器地址，谁先返回就用谁的（竞速机制）：

| 端点 | 地址 |
|------|------|
| 主端点 | `bing.kaininx.workers.dev` |
| 备用端点 | `bing.biturl.top` |

- 两个请求同时发出，取最快响应的那个
- 拿到结果后立刻取消另一个请求（省带宽）
- 最长等 8 秒，超时就放弃
- 请求参数里带「市场代码」——根据当前语言决定去哪国 Bing 拿图（中文→中国 Bing，日文→日本 Bing，其他→美国 Bing）

### 5.2 缓存机制

拿到的 Bing 图片不是每次都重新下载的：

- **同一天内**：URL 一样 → 不再下载，直接用本地缓存的 blob
- **URL 一样但 blob 丢了**（比如清理了浏览器数据）→ 重新下载
- **URL 不一样**（新一天的图）：下载 → 存到本地 → 更新记录

缓存存在 localStorage（记录这张图的 URL、日期、来源）和 IndexedDB（图片文件本体）。

### 5.3 后台预缓存

即使用户当前在用「本地轮换」模式，系统也会在后台静默检查今天的 Bing 图有没有缓存好。这样用户随时切回 Bing 模式都不用等下载。

### 5.4 网络失败处理

- 双端点都超时（8 秒没响应）→ 保持底层的旧图/缩略图，用户不会看到白屏
- Bing 返回了 URL 但下载图片失败 → 保持旧图，等下次新标签页再试
- 所有路径都失败 → 不报错，底层至少还有 `preload.js` 写的缩略图

## 六、本地图片轮换

### 6.1 基本规则

- 最多存 12 张
- 列表 `ptab_img_order` 决定了轮换顺序（谁在前谁先播）
- 每次打开新标签页，索引 +1，取模列表长度，播下一张
- 列表为空时，自动走 Bing 路径

### 6.2 缩略图管理

每张本地图片存一份缩略图（宽度 640px 的 JPEG），用于预加载环节。缩略图在以下时机生成：
- 用户上传图片时
- 壁纸切换时发现某张图缺缩略图（自愈）

**缩略图体积控制：** 宽度固定 640px，JPEG 质量 0.55，单张大约 40-50KB。12 张全存大约占 500-600KB，在 localStorage 5MB 配额内安全。

### 6.3 轮换索引与预加载的竞态处理

这是一个容易出问题的点：本地模式下，每次打开新标签页索引会 +1。如果两个新标签页快速连续打开，有可能：
- 标签页 A 读到索引 0，开始展示图片
- 标签页 A 还没写完「下一张的缩略图」
- 标签页 B 打开了，读到索引 1，但「下一张的缩略图」可能还是旧值

解决办法：**索引递增后立刻同步写预览缩略图**（在异步展示图片之前），确保索引和预览缩略图同时更新。这样无论标签页打开多快，`preload.js` 读到的索引和预览缩略图永远是对应的。

如果下一张缩略图还不存在（新上传的图片还没来得及生成缩略图），主动把 `ptab_preview_thumb` 删掉，让预加载走 fallback 路径，避免展示了错误的缩略图。

### 6.4 Blob MIME 类型恢复

图片存 IndexedDB 后，有些浏览器取出来时文件类型（MIME）会丢失。系统取出 blob 后检查：如果类型为空且存了 MIME 信息，就用原始 MIME 重建一个正确的 Blob。

## 七、缩略图生成

任何需要缩略图的地方（Bing 缓存写缩略图、本地图片上传写缩略图、自愈补缩略图），都走同一个流程：

1. 拿到图片（可能是 URL 字符串，也可能是已经加载到内存的图片对象）
2. 如果传的是内存中的图片对象，直接用——省一次网络请求和解码
3. 创建 Canvas，把图片缩到 640px 宽，等比缩放高度
4. 导出为 JPEG base64，质量 0.55
5. Canvas 长宽归零——提示浏览器释放显存
6. 返回 `url(data:image/jpeg;base64,...)` 格式的字符串

**失败处理：** 图片损坏、跨域被拦、Canvas 导出失败 → 返回空，不写缩略图。下次需要时重新尝试。

---

## 八、Blob URL 的生命周期

每次展示本地壁纸或缓存的 Bing 壁纸时，会把 Blob 转成 Blob URL（`blob:...`）。这个 URL 会设到底层/前层的背景里。

**释放时机：** 新壁纸过渡完成后（550ms），旧 Blob URL 不再被任何 DOM 引用，立即调用 `URL.revokeObjectURL()` 释放。不是 Blob URL（比如直接显示 Bing 原始 HTTPS URL）时，也要检查有没有旧的 Blob URL 需要释放。

**为什么不在过渡前释放：** 过渡期间底层可能还在显示旧图，提前释放会导致旧图闪烁消失。

## 九、壁纸信息显示

设置面板里有一行文字显示当前壁纸的信息：

| 模式 | 显示内容 |
|------|---------|
| Bing | "🌄 Bing 每日壁纸"（多语言） |
| 本地 | "📷 本地壁纸"（多语言） + 图片名称可选 |

文字在壁纸切换完成后更新。

## 十、状态转换图

```
         ┌──────────┐     上传第一张      ┌──────────┐
         │  Bing    │ ─────────────────→  │  本地    │
         │  每日    │                      │  轮换    │
         └──────────┘ ←───────────────── └──────────┘
                ↑      删光/重置/切回            │
                └───────────────────────────────┘

加载优先级（每次打开新标签页）：
    本地轮换（有图片）→ Bing 本地缓存（今天的）→ Bing 网络获取 → 保持预加载缩略图
```

## 十一、与其它模块的关系

| 依赖谁 | 依赖什么 |
|--------|---------|
| 数据存储 | 所有的读写删操作，idbPut/Get/Delete、loadOrder/saveOrder 等 |
| 语言系统 | 当前语言 → 决定 Bing 市场代码；壁纸信息文本（多语言） |
| 本地图库 | 上传、删除、重置操作由本地图库触发 |

| 被谁依赖 | 怎么依赖 |
|---------|---------|
| 预加载（preload.js） | 读缩略图、模式、索引 |
| 本地图库 | 上传图片时调用壁纸系统的展示功能；删除时触发模式切换 |
| 应用启动 | 启动时加载壁纸是第一步 |

## 十二、时序约束

**文件加载顺序（绝对不能乱）：**
```
HTML 解析 → #wallpaperBack 出现 → preload.js 同步执行（写底层背景）
→ #wallpaperFront → languages.js → newtab.js（异步加载壁纸）
```

**双层过渡时序（毫秒级）：**
```
0ms:    新图设到前层（opacity 为 0，用户看不见）
~16ms:  浏览器提交前层新背景（第一个 rAF）
~32ms:  前层开始淡入（第二个 rAF，触发 CSS transition）
32-532ms: 动画进行中
550ms:  动画结束，同步到底层，清空前层
```

**本地模式预览时序（关键）：**
```
0ms:  索引递增 → 立即同步写下一张的预览缩略图（同步操作）
~1ms: 开始异步展示当前壁纸
```
