## 零、这插件是干嘛的

装了之后，浏览器的新标签页变成：**一张全屏壁纸 + 一个搜索框**。没有新闻、广告、快捷链接。

它还附带这些能力：

- 每天自动换一张 Bing 美图
- 上传自己的图片做壁纸，最多 12 张，每次打开自动轮播
- 支持 16 种语言
- 可以在 Google / Bing / 百度 / DuckDuckGo 之间切换搜索（网页模式）
- 既能在 Chrome / Edge 里当插件用，也能部署成普通网页直接访问

## 一、打开一个新标签页 —— 慢镜头回放

你按 `Ctrl+T`，浏览器开始加载 PlainTab。整个过程分 4 个阶段：

```
按 Ctrl+T
  │
  ▼
第 0 步：加载文件（index.html + CSS + 3 个 JS）
  │
  ▼
第 1 步：preload.js 瞬间贴图（防白屏）★ 最关键
  │
  ▼
第 2 步：languages.js 加载翻译表
  │
  ▼
第 3 步：newtab.js 主程序启动（壁纸 + UI + 搜索）
```

三个 JS 文件的加载顺序**不能乱**，原因往下看。

---

### 第 1 步（最关键）—— 防白屏

> 🎯 **要解决的问题**

浏览器显示一张网络图片，需要先下载。这个"下载空窗期"里，屏幕是白的（或灰的），大概持续几百毫秒到一两秒。眼睛很不舒服。

> 💡 **PlainTab 的做法：压图缓存**

上一次关闭标签页前，插件偷偷把当前壁纸"拍照"成一张只有 **640 像素宽**的压缩小图（大概 30~60KB），存进浏览器的 localStorage（一个能瞬间读取的小仓库）。

下一次打开新标签页，`preload.js` 在第一毫秒就把这张小图贴在背景上。用户看到的从始至终都有画面，没有白屏。

然后，高清大图在后台悄悄加载、渐变替换掉小图。用户只看到一次平滑的过渡，不是"白→闪→图"。

> 🏠 **类比**

电影院幕布本身就是深色的，灯灭了也不刺眼。PlainTab 相当于你上次离场时，把幕布涂成了壁纸的颜色。下次进来，幕布上有画，不是白墙。

> 📐 **关键数字：为什么是 640px 宽**

| 宽度 | 文件大小 | 效果 |
|:--|:--|:--|
| 200px | ~5KB | 全是马赛克，挡不住白屏 |
| **640px** | **~40KB** | **够清晰挡住白屏，文件又够小** |
| 1920px | ~300KB | 太大，localStorage 放不下 |

640 是平衡点：文件小到 localStorage 能轻松容纳、清晰度又足够撑场面。

---

### 第 2 步 —— 翻译表上膛

`languages.js` 做的事情极度简单：定义了一个 16 种语言的翻译字典。

```
"搜索" →
  中文(简体)：在 Web 中搜索…
  English：Search the Web…
  日本語：Web を検索…
  ...（共 16 种）
```

> 📐 **为什么必须单独一个文件**

它必须在 `newtab.js`（主程序）之前加载。如果顺序反了，主程序启动时找不到翻译表，所有文字都会显示成英文的 key 名（比如 `searchPlaceholder`），满屏都是代码变量名。

## 二、主程序拆解

### 🧭 环境检测：我是插件还是网页？

程序醒来第一件事：判断自己的身份。

```
chrome.runtime.id 存在？ → 我是插件模式
                      不存在？ → 我是网页模式
```

同一个程序，两种身份，行为不同：

| | 插件模式 | 网页模式 |
|:--|:--|:--|
| 搜索方式 | 用浏览器自带搜索引擎 | 用户自选（4 个） |
| 搜索图标 | 静态放大镜 🔍 | 彩色引擎图标 |
| 点击切换引擎 | ❌ | ✅ |
| 引擎下拉框 | 隐藏 | 显示 |

> 📐 **为什么插件模式不给选引擎**

Chrome 应用商店有一条"单一用途政策"：每个插件只能做一件事。PlainTab 注册的用途是"替换新标签页"。如果还提供多引擎搜索，审核可能被拒。所以插件模式老老实实用浏览器默认搜索引擎。

---

### 🗄️ 两个存储仓库

浏览器给了两种存数据的方式，PlainTab 各用各的：

**① localStorage — 像桌上的便签纸**

伸手就拿到，速度极快，**同步读取**。但空间小（5-10MB）。

| 存什么 | 为什么放这里 |
|:--|:--|
| `bing_thumb`（Bing 当前缩略图） | preload.js 要瞬间读，不能等 |
| `local_thumbs`（本地壁纸缩略图数组） | preload.js 轮播用，不能等 |
| `ptab_local_index`（轮播指针） | 每次打开翻到下一张，不能等 |
| 语言、壁纸模式、搜索设置 | 启动时立刻要知道，不能等 |
| `ptab_bing_meta`（Bing URL/日期） | 缓存去重用 |

**② IndexedDB — 像书房的文件柜**

空间大（可达几百 MB），但打开要花几十毫秒，**异步读取**。

| 存什么 | 为什么放这里 |
|:--|:--|
| `bing`（Bing 壁纸原图 Blob） | 几百 KB，localStorage 放不下 |
| `local_images`（用户上传的壁纸数组） | 每张几百 KB~几 MB，最多 12 张 |

> 🔧 **MIME 类型修复**

把图片存进 IndexedDB 再取出来，有时浏览器会"忘记"这文件是 jpg 还是 png（Blob.type 变成空字符串）。所以存的时候顺手记下 `mime` 字段（如 `image/jpeg`），取的时候手动重建一个带正确类型的 Blob。

---

### 💓 双层图层 —— 壁纸系统的心脏

这是整个插件最核心的架构。

> 🏠 **类比：舞台剧换景**

好导演用双层幕布换景：
- **后层**（实色幕布）：永远有画面，观众始终能看到东西
- **前层**（半透明幕布）：平时透明。换景时悄悄布好新景 → 前层慢慢变不透明 → 新景露出 → 后层"凝固"成新画面 → 前层恢复透明

从头到尾，观众没看到过一刻空白。

---

在代码里就是两个 `<div>`：

```
┌──────────────────────────────┐
│  #wallpaperFront (上层)       │  ← 默认透明，淡入用
│  opacity: 0 → 1 过渡 0.5s    │
├──────────────────────────────┤
│  #wallpaperBack (下层)        │  ← 永远显示，绝对不空
│  始终有一张图                 │
└──────────────────────────────┘
```

---

**展示新壁纸的 8 个步骤：**

```
Step 1  后台下载新图（用户看不见）
         │
Step 2  设到 front 层上
         │
Step 3  等 2 帧，让浏览器准备好
         │
Step 4  front 层从透明渐变到不透明（0.5 秒）
         │  用户看到的：旧图 → 平滑过渡 → 新图
Step 5  再等 50ms，确保动画播完
         │
Step 6  把新图"固定"到 back 层（直接赋值）
         │
Step 7  清空 front 层，恢复透明
         │
Step 8  生成缩略图存 localStorage（给下次 preload.js 用）
```

---

> 📐 **为什么多等 50ms（550ms 而不是 500ms）**

CSS 过渡设为 500ms。有些浏览器在极端情况下可能延迟几毫秒才播完。多等 50ms 是安全余量——确保动画 100% 结束了再做下一步。

> 📐 **为什么要"等 2 帧"（两次 requestAnimationFrame）**

如果你在同一毫秒内改了背景图又改了透明度，浏览器会把两次修改"打包处理"——结果它看到的是"直接变不透明"，跳过了过渡动画。强制分两帧：第一帧应用背景图，第二帧启动透明度动画。动画才能正常播放。

---

### 🌐 Bing 壁纸是怎么拿到的

程序启动后，**同时**读取两个来源（并行，不等串行）：

```
同时读取
├── IDB 里的本地壁纸
└── IDB 里的 Bing 缓存
         │
         ▼
    按优先级决策：

优先级 ①  用户选了本地壁纸，且有文件 → 用本地
优先级 ②  今天的 Bing 图已缓存 → 直接用，不走网络
优先级 ③  没任何缓存 → 走网络下载
```

> 📐 **为什么并行读取**

串行的话，两个操作排队等，耗时累加。并行是同时出发，取最慢的那个的时间。大约快一倍。

---

**优先级 ③ 的网络流程：**

```
有今天的 URL 但没有下载好的文件？
  → 先用 URL 快速设到 back 层显示，后台下载并存 IDB

连 URL 都没有？
  → 同时请求两个 API（竞赛！），各给 8 秒超时
  → 谁先返回就用谁
  → 另一个的结果直接丢弃
  → 下载 → 展示 → 缓存
  → 两个都挂了 → 保持旧图不动
```

> 📐 **为什么同时请求两个 API，而不是一个一个试**

两个 API 就是两条路。串行的话，第一条路不通要等很久才知道。并行的话，快的那条路瞬间就到。JSON 响应只有几百字节——多走一条路几乎零代价，却换来了不管在哪都是最优速度的体验。

---

**去重机制：同一天不重复下载**

Bing 每天只换一次图。用户一天开 10 次新标签页，每次都下载同一张图？太浪费了。

```
下载前先请求 JSON（只含图片 URL，不含图片本身，很轻量）
  → 对比和上次存的 URL 是否一样
  → 一样 → 跳过，不下载
  → 不一样 → 新图，下载
```

---

### 🔇 后台静默缓存

即使用户选了本地壁纸，插件也会**悄悄在后台**下载今天的 Bing 图并存好。

如果哪天用户点了"恢复 Bing 每日壁纸"→ 图已经在缓存里了，秒切换。不用等下载。

> 📐 **为什么提前下载**

从用户点"恢复"到看到图，如果中间还要等下载，体验就差一截。提前缓存好，切换就是瞬间。

---

### 📸 缩略图的生成

```
创建画布 → 把高清图画上去（缩小到 640px 宽）
  → 导出为 JPEG（质量 55%）
  → 转成 CSS 格式 → 存进 localStorage
```

> 📐 **为什么 JPEG 而不是 PNG**

同样 640px 宽的图：JPEG 质量 55% 约 30-60KB，PNG 约 200-500KB。localStorage 空间宝贵，JPEG 碾压。

> 📐 **为什么质量 55%**

- 90%+ ：文件太大，localStorage 可能撑爆
- 20% ：明显色块模糊，挡白屏效果差
- **55%** ：肉眼几乎看不出损失，文件够小

---

### 🖱️ UI 交互：鼠标追踪

PlainTab 的 UI 哲学：**鼠标不动 — 只有壁纸。鼠标动 — 需要的才出现。**

#### 右上角两个按钮

```
设置按钮（齿轮 ⚙）：始终 45% 半透明，鼠标悬停 → 100% 亮起
语言按钮（地球 🌐）：默认隐藏，鼠标到右上角 → 出现
```

**显示/隐藏规则：**

```
鼠标进入右上角（右 180px × 上 130px 的区域）
  → 两个按钮都显示

鼠标离开
  → 等 400ms → 隐藏
  → 但如果面板正开着 → 不隐藏
```

> 📐 **为什么设置按钮半透明、语言按钮完全隐藏**

设置按钮是"锚"——用户需要感知到"这有个可以点的东西"。半透明齿轮不打扰沉浸感，又暗示存在。语言切换频率极低，全隐藏更干净。

> 📐 **为什么等 400ms 才隐藏**

鼠标在屏幕上快速移动时，可能"划过"右上角又立刻离开。如果瞬间隐藏，按钮闪一下就没了。等 400ms 确认"真的离开了"再藏。这叫防抖。

---

#### 搜索栏的三种模式

| 模式 | 行为 |
|:--|:--|
| 鼠标触发 | 鼠标到屏幕中央 → 出现；离开 150ms → 隐藏 |
| 常驻显示 | 永远在 |
| 始终隐藏 | 永远不在（极简主义者的最爱） |

额外保护：如果光标正在输入框里打字 → 搜索栏绝不消失。

### ⚙ 设置面板

从右上角齿轮按钮滑出：

```
┌─ 壁纸信息：Bing 每日 / 本地壁纸
├─ [选择本地壁纸]
├─ [恢复 Bing 每日壁纸]
├─ ⚙️ 高级选项（折叠）
│   ├─ 搜索栏模式：鼠标触发 / 常驻 / 隐藏
│   ├─ 图标不透明度：滑块 + 数字框
│   ├─ 搜索引擎：下拉选择
│   └─ [恢复默认]
└─ GitHub 链接
```

**关闭面板的 4 种方式：**

点击齿轮 / 鼠标离开 500ms / 点面板外面 / 按 Esc

> 📐 **透明度为什么滑块 + 数字框都有**

滑块 = 快速"大概拖一下"。数字框 = 精确输入 0.37。两种方式互补。

---

### 🌐 语言面板

结构和设置面板一样，内容是 16 种语言按钮。当前语言高亮。

切换语言后 → 刷新所有文字 → 关面板。壁纸下次打开新标签页时重拉（因为不同地区的 Bing 图不同）。

> 📐 **为什么不立刻刷新壁纸**

用户可能只是试一下日语马上切回来。立刻下载两次就浪费了。延迟到"下次打开"才刷新，节能。

### 📁 本地上传壁纸（多图 + 轮播）

支持上传最多 **12 张**本地壁纸。每次打开新标签页，自动轮播到下一张。

**上传流程：**

```
点"选择本地壁纸" → 文件选择框打开（可多选）
  → 每个文件逐一：
      A. 去重：文件名+大小都相同 → 跳过
      B. 生成缩略图（Canvas 缩小到 640px 宽）
      C. 原图存进 IndexedDB（local_images 数组）
      D. 缩略图存进 localStorage（local_thumbs 数组）
      E. 两个数组严格对齐：第 N 个 blob ↔ 第 N 个缩略图
```

> 📐 **为什么缩略图要单独存一个数组**

`preload.js` 首帧需要瞬间读取。IndexedDB 是异步的、需要几十毫秒才能打开。localStorage 是同步的、瞬间读。所以缩略图放在 localStorage 能让首帧不等待。

**轮播机制：**

```
每次打开新标签页：
  ptab_local_index += 1
  取 local_images[index % 总数]
  取 local_thumbs[index % 总数] → 首帧瞬间显示
  同时高清大图在后台加载并渐变替换
```

**批量导入：**

文件选择框支持一次选多张（`multiple` 属性）。选了 20 张，只存前（12 - 已有）张，多余的自动忽略。第一张存完立刻设成壁纸展示，其余静默存库。

**画廊：**

打开设置面板时，本地模式下显示一个 3×4 的缩略图网格。每张缩略图右上角有删除按钮（×）。满 12 张时 "+" 按钮自动隐藏。

删除最后一张 → 自动切回 Bing 每日壁纸模式。

**恢复 Bing 模式：**

点"恢复 Bing 每日壁纸"按钮。如果本地图 ≥ 2 张，会弹确认框防误操作。确认后清空所有本地数据，自动切回 Bing 每日壁纸。

## 三、插件 vs 网页，两种模式

| | 插件 | 网页 |
|:--|:--|:--|
| 替代新标签页 | ✅ 自动 | 需手动设首页 |
| 搜索引擎 | 浏览器默认 | 自选 4 个 |
| 检测语言 | 读浏览器 UI 语言 | 读系统语言 |
| 引擎图标 | 灰色放大镜 | 彩色品牌图标 |
| CWS 合规 | ✅ 单一用途 | 不受限 |

**404.html** 是给 静态托管平台用的兜底页——访问不存在的路径时，5 秒倒计时自动跳回首页。

## 四、设计哲学

### 零依赖

不引入任何第三方库。全部手写原生 JS。

> 好处：加载极快（没有框架要下载）、没有供应链漏洞、代码审计一眼看到底。

### 渐进增强

先展示（缩略图快速上墙），再完善（高清图后台替换）。UI 也是先默认值，用户再个性化。

> 好处：首次打开不用等，弱网/离线也不白屏。

### 静默失败

错误不弹窗、不报错、不崩溃。Bing API 挂了 → 旧图顶着。缩略图生成失败 → 壁纸照样显示。IDB 写入失败 → 下次重试。

> 好处：用户几乎感知不到出错。壁纸上永远有东西。

### 用户主权

所有设置持久化保存，永远不会被重置。每次打开新标签页 → 和上次离开时一模一样。

## 六、优化建议

**1. 缩略图分辨率自适应**
> 现状：缩略图写死 640px 宽。4K 屏幕上略糊，小屏幕上又浪费空间。
> 改法：根据屏幕实际分辨率动态调整缩略图宽度。

**2. 预加载下张壁纸缩略图**
> 现状：本地多图轮播时，每次打开新标签页才生成当前壁纸的缩略图。如果缩略图缺失（崩溃/数据错位），该次打开会有短暂白屏，靠自愈机制下次修复。
> 改法：当前壁纸展示后，后台提前为下一张轮播壁纸生成缩略图。
